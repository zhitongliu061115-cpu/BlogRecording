# BlogRecording 社区治理指南 (Governance)

欢迎来到 **BlogRecording** 开源社区！

本项目秉持 **“Community over Code（社区优先于代码）”** 的核心理念。为了建立一个开放、透明、可持续的开源社区，我们起草了本治理文档，明确项目的角色分工、贡献者的成长路径以及社区的决策机制。

## 1. 社区角色与职责 (Role Matrix)

即使在项目的早期阶段，我们也定义了清晰的治理角色，以确保项目的规范运转。目前部分角色可能由核心创立者兼任，但我们时刻欢迎社区成员通过贡献接管这些角色。

*   **Maintainer (维护者)**
    *   **职责：** 把握项目整体战略方向，负责版本发布周期管理，拥有主分支的最终合并权限。
    *   **产生方式：** 现阶段由项目发起人担任，后续从核心提交者中选举产生。
*   **Reviewer (评审者)**
    *   **职责：** 负责日常 Issue 的分类与响应，审查外部提交的 Pull Request，确保代码和文档符合质量标准。
    *   **产生方式：** 由长期活跃的 Contributor 提名，Maintainer 批准。
*   **Compliance (合规与安全管理员)**
    *   **职责：** 审查项目依赖包的许可证兼容性（生成 SBOM），监控潜在的安全漏洞，处理安全相关的举报。
    *   **产生方式：** 现阶段由 Maintainer 兼任。

## 2. 贡献者成长路径 (洋葱模型)

我们为每一位社区参与者规划了清晰的晋升路线。你的每一次参与，都会增加你在社区的“社交资本”和影响力：

1.  **Watcher (关注者)：** 关注项目动态，为项目点赞 (Star) 和分享。
2.  **User (活跃用户)：** 在实际场景中使用本项目，积极在 Issue 区提交 Bug 反馈、需求建议或参与功能讨论。
3.  **Contributor (贡献者)：** 成功合并过至少 1 个 Pull Request（包括代码优化、Bug 修复、文档完善、测试用例补充等）。
4.  **Committer (核心提交者)：** 持续贡献高质量代码或文档，深度参与技术讨论。获得 Committer 身份后，将被授予仓库的直接提交和合并权限（Write Access）。
5.  **TC (技术委员会委员) / PMC：** 深入了解项目架构，主导核心模块开发或重大架构设计。参与制定项目 Roadmap 及关键技术选型。

## 3. 共识与决策机制

为了在“保证迭代效率”与“尊重社区意见”之间取得平衡，我们在不同阶段采用不同的共识机制：

### 3.1 现阶段：仁慈的独裁者 + 懒人共识
*   **仁慈的独裁者 (Benevolent Dictator)：** 在项目初期阶段，为了保证产品愿景的第一性原理和开发效率，项目的最终决策权保留在项目发起人（Maintainer）手中。但这并不意味着独断专行，所有决策都会在 Issue 或 Discussion 中公开透明地讨论。
*   **懒人共识 (Lazy Consensus)：** 对于日常的功能迭代、非破坏性的代码修改和文档更新，我们采用“懒人共识”。任何人都可以在 Issue/PR 中提出建议，如果 **72小时内** 没有人提出明确且合理的反对意见，该提议即视为自动通过并执行。这旨在降低沟通成本，鼓励行动。

### 3.2 规模化后：精英治理与委员会投票
当我们的 Committer 数量达到一定规模后，项目的治理将自动过渡到 PMC (项目管理委员会) 模式。重大决策（如架构重构、API 破坏性变更、变更许可证等）将通过 TC/PMC 成员的投票产生（遵循 Apache 社区的投票规范，如 +1, 0, -1）。

## 4. 开发者原创声明 (DCO)

本项目无需签署繁琐的 CLA。为确保知识产权的合规性，我们采用轻量级的 **DCO (Developer Certificate of Origin)** 机制。

当您提交 PR 时，请确保您的 Commit 信息包含 `Signed-off-by` 签名。使用 `git commit -s -m "您的提交信息"` 命令即可自动添加。这代表您确认您有权提交该代码，并同意以本项目的许可证（如 Apache 2.0 / MIT）将其开源。

## 5. 行为准则
我们在所有的交互中严格遵守 [CODE_OF_CONDUCT.md](./CODE_OF_CONDUCT.md)。任何违反行为准则的情况都可以向 Maintainer 报告。

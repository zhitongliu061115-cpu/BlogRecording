# SDD 与 Harness 开发管理规范

本文用于规范项目如何按 SDD（规格驱动开发）和 harness（自动化验证门禁）进行开发、评审、测试和交付。

核心原则：

- 先写清楚规格，再写代码。
- 每个变更都有明确目标、范围、任务和验收标准。
- 代码能不能合并，不靠感觉，靠自动化检查和评审结果。
- 需求、实现、测试、发布记录要能互相追溯。

## 1. 目录约定

本项目使用以下目录管理开发过程：

```text
openspec/
  config.yaml          OpenSpec 配置
  changes/             待开发或开发中的变更
  specs/               已稳定的能力规格

docs/
  Git 使用规范.md
  SDD 与 Harness 开发管理规范.md
```

建议每个变更在 `openspec/changes/` 下使用一个独立目录：

```text
openspec/changes/<change-id>/
  proposal.md          变更说明
  design.md            设计说明，可选
  tasks.md             任务清单
  specs/               本次变更涉及的规格增量
```

`change-id` 使用短横线命名，例如：

```text
add-recording-history
fix-summary-empty-response
improve-model-loading-check
```

## 2. SDD 工作流程

### 第一步：提出变更

任何功能、修复、重构、权限、存储、网络、模型相关改动，都先创建变更目录。

`proposal.md` 至少写清：

- 背景：为什么要做。
- 目标：做完后用户或系统能得到什么。
- 范围：本次做什么，不做什么。
- 影响：会影响哪些页面、模块、数据或权限。
- 验收：什么结果算完成。

示例：

```markdown
# 新增录音历史筛选

## 背景
历史记录变多后，用户很难快速找到失败或已完成的录音。

## 目标
用户可以按录音状态筛选历史记录。

## 范围
- 增加状态筛选入口。
- 支持全部、录音中、已完成、失败。

## 不做
- 不做关键词搜索。
- 不改记录存储结构。

## 验收
- 用户切换筛选项后，列表只显示对应状态的记录。
- 默认显示全部记录。
```

### 第二步：写规格

规格描述系统应该表现成什么样，不写具体代码实现。

规格要包含：

- 用户可见行为。
- 输入、输出和状态变化。
- 失败场景。
- 兼容要求。
- 测试可验证的验收标准。

写法建议：

```markdown
## Requirement: 历史记录状态筛选

### Scenario: 默认显示全部记录
Given 用户进入历史页面
When 未选择筛选条件
Then 页面显示全部录音记录

### Scenario: 筛选失败记录
Given 历史页面存在失败和已完成记录
When 用户选择“失败”
Then 页面只显示失败记录
```

### 第三步：拆任务

`tasks.md` 按可落地步骤拆分，不要写成空泛口号。

推荐格式：

```markdown
# Tasks

- [ ] 更新历史页面状态筛选 UI。
- [ ] 在 ViewModel 中增加筛选状态。
- [ ] 根据筛选状态计算展示列表。
- [ ] 补充筛选逻辑单元测试。
- [ ] 手动验证全部、已完成、失败三种筛选结果。
```

任务要求：

- 每个任务最好能在半天内完成。
- 每个任务都能明确判断完成或未完成。
- 涉及行为变化时必须包含测试任务。

### 第四步：实现

实现时只做当前变更目录里定义的内容。

要求：

- 不顺手做无关重构。
- 不改未在规格里说明的行为。
- 如果实现中发现规格不完整，先补规格，再继续写代码。
- 如果实际方案与 `design.md` 不一致，更新设计说明。

### 第五步：验证

实现完成后必须跑 harness 检查。检查结果写进 PR。

最少检查：

```powershell
git status --short
git diff --check
.\gradlew.bat testDebugUnitTest
.\gradlew.bat assembleDebug
```

如果涉及 UI：

- 手动验证关键页面。
- 必要时补充 Compose UI Test。

如果涉及录音、权限、网络、模型、密钥：

- 真机或模拟器验证主流程。
- 验证失败提示。
- 确认没有泄露敏感信息。

### 第六步：合并与归档

PR 合并前确认：

- 规格已更新。
- 任务已完成。
- 测试和构建通过。
- PR 描述包含验证结果。
- Reviewer 认可行为变化。

合并后，如果变更已经稳定：

- 将规格增量同步到 `openspec/specs/`。
- 归档或关闭 `openspec/changes/<change-id>/`。
- 保留关键决策和验收结果，方便后续追溯。

## 3. Harness 门禁

harness 是项目的自动化检查门禁。它的作用是保证每次合并前都经过同一套验证。

### 必选门禁

每个 PR 都必须通过：

- Git 状态检查：确认没有误提交文件。
- 空白检查：确认没有明显格式错误。
- 单元测试：验证核心逻辑。
- Debug 构建：确认项目能正常编译。

推荐命令：

```powershell
git status --short
git diff --check
.\gradlew.bat testDebugUnitTest
.\gradlew.bat assembleDebug
```

### 按变更类型增加门禁

| 变更类型 | 额外检查 |
| --- | --- |
| UI 变更 | 手动验证页面，必要时补 Compose UI Test |
| 数据存储变更 | 验证旧数据读取、异常数据处理 |
| 录音相关变更 | 真机验证开始、停止、权限拒绝、异常中断 |
| 网络相关变更 | 验证超时、401、429、非 2xx、空响应 |
| 模型相关变更 | 验证模型缺失、模型路径、初始化失败 |
| 安全相关变更 | 检查密钥不入库、日志不打印敏感信息 |

### 门禁失败处理

- 测试失败：先修复，再重新跑完整检查。
- 构建失败：不能合并。
- 环境失败：在 PR 写明原因、机器环境和替代验证方式。
- 无法验证的风险点：必须明确写在 PR 里，交给 Reviewer 判断。

## 4. PR 模板

每个 PR 建议使用以下结构：

```markdown
## 变更说明
- 

## 关联规格
- openspec/changes/<change-id>/

## 验证结果
- [ ] git status --short
- [ ] git diff --check
- [ ] .\gradlew.bat testDebugUnitTest
- [ ] .\gradlew.bat assembleDebug

## 风险
- 

## 安全与隐私
- [ ] 未提交 API Key、Token、证书、签名文件
- [ ] 未提交真实录音、用户数据、日志
- [ ] 新增权限、网络、存储或模型变更已说明原因
```

## 5. 角色职责

### 需求提出人

- 说明要解决的问题。
- 确认验收标准。
- 验收最终结果。

### 开发者

- 创建或更新 SDD 变更。
- 按规格实现。
- 补充测试。
- 在 PR 中写清验证结果。

### Reviewer

- 检查规格是否清楚。
- 检查实现是否超出范围。
- 检查测试是否覆盖主要风险。
- 检查是否有敏感信息或大文件误提交。

### 维护者

- 维护 `main`、`develop` 分支质量。
- 决定变更是否可以合并。
- 合并后推动规格同步和归档。

## 6. 开发纪律

- 不允许先写一大堆代码，最后再补规格。
- 不允许需求没定就直接合并。
- 不允许跳过失败的测试强行合并。
- 不允许把敏感信息、真实录音、日志和本机配置提交到仓库。
- 不允许一个 PR 混合多个无关目标。
- 可以小步快跑，但每一步都要有规格、任务和验证结果。

## 7. 推荐执行顺序

一次标准变更按这个顺序执行：

```text
提出想法
-> 创建 openspec/changes/<change-id>/
-> 写 proposal.md
-> 写 specs 增量
-> 写 tasks.md
-> 评审规格
-> 创建开发分支
-> 实现任务
-> 跑 harness 检查
-> 提交 PR
-> Review
-> 合并
-> 同步规格
-> 归档变更
```
